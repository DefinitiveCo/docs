The Orders WebSocket streams a funder's open orders in real time. Subscribe and you receive a snapshot of the funder's current open orders, then an `update` whenever an order changes or a new one is placed. Use it instead of polling [`GET /orders`](/api-reference/flash/list-orders) or [`GET /orders/{orderId}`](/api-reference/flash/get-order) when you need live order state.

**Endpoint:** `wss://flash.definitive.fi/v1/ws`

## Channels

| Channel | What it delivers | Auth required |
| --- | --- | --- |
| `orders` | A funder's open orders: a `snapshot` on subscribe, then an `update` when an order changes or a new one is placed | Yes |
| `heartbeats` | A keep-alive frame once per second | Yes |

## Authenticating

Both channels require authentication. Include your Flash API key as `apiKey` in every `subscribe` message. A read-scoped key works, since the stream is read-only. Get a key from [Become an Integrator](/getting-started).

## Working with the WebSocket

- Both channels run over the same connection. Open one socket, then subscribe to `orders` and `heartbeats` on it.
- Send your `subscribe` message as soon as the connection opens.
- One connection streams one funder. Open a separate connection for each funder you track.
- The per-key connection limit is >50. If you need it raised, reach out.
- Also subscribe to the [`heartbeats`](#heartbeats-channel) channel so the connection stays open when order updates are sparse. Idle connections with no traffic are closed after about 60 seconds.
- On disconnect, reconnect with capped backoff and jitter, then re-subscribe. Every subscribe returns a fresh snapshot.
- On `UNAUTHORIZED`, stop reconnecting and check your API key; reconnecting with the same key will not authenticate. Other closes, including `CONNECTION_LIMIT`, are safe to retry with backoff.
- If the stream is unavailable (`STREAMING_UNAVAILABLE`), fall back to polling [`GET /orders`](/api-reference/flash/list-orders).

## Orders channel

### Subscribe

```json
{ "channel": "orders", "type": "subscribe", "funderAddress": "0x…", "apiKey": "dpka_…" }
```

The server confirms with an `ack` listing your active channels:

```json
{ "channel": "subscriptions", "type": "ack", "subscriptions": ["orders"] }
```

### Snapshot

Once, right after you subscribe, you receive the funder's current open orders:

```json
{ "channel": "orders", "type": "snapshot", "orders": [ /* FlashOrderUpdate[] */ ] }
```

### Update

When an order changes, or a new order is placed, you receive an `update` carrying just that order:

```json
{ "channel": "orders", "type": "update", "orders": [ /* FlashOrderUpdate */ ] }
```

### Applying updates

- Seed your local state from the `snapshot`, keyed by `orderId`.
- Each `update` is the order's full current state. Replace your stored copy by `orderId`, or add it if you have not seen the order yet (a newly placed order arrives this way).
- An `update` also fires when an order reaches a terminal status. Treat any status other than `ORDER_STATUS_PENDING`, `ORDER_STATUS_ACCEPTED`, or `ORDER_STATUS_PARTIALLY_FILLED` as terminal, and drop it from an open-orders list.

### Order fields

`snapshot` and `update` carry the same `FlashOrderUpdate` shape:

| Field | Type | Notes |
| --- | --- | --- |
| `orderId` | `string` | Stable key. Merge and dedupe on this. |
| `orderType` | `market`, `limit`, `twap`, `stop`, `stop-loss`, `take-profit`, `bracket` | |
| `side` | `buy` or `sell` | |
| `status` | `string` | Active: `ORDER_STATUS_PENDING`, `ORDER_STATUS_ACCEPTED`, `ORDER_STATUS_PARTIALLY_FILLED`. Any other value is terminal (for example `ORDER_STATUS_FILLED`, `ORDER_STATUS_CANCELLED`, `ORDER_STATUS_REJECTED`, `ORDER_STATUS_TERMINATED`); drop it from an open list. See [Order statuses](/order-statuses). |
| `closeReason` | `string` or `null` | For example `REASON_USER_REQUESTED` or `REASON_FULLY_FILLED`. `null` while active. |
| `funderAddress` | `string` | The funder wallet whose orders you subscribed to. |
| `targetAsset` | `{ id, name, address, ticker, chain }` | The traded asset. |
| `contraAsset` | `{ id, name, address, ticker, chain }` | The counter asset, spent on buys and received on sells. |
| `qty` | `string` (decimal) | Amount being spent, in the asset's normalized units. |
| `filled` | `{ targetAmount, contraAmount }` or `null` | `null` means nothing filled yet. |
| `limitNotionalPrice` | `string` or `null` | USD limit price for the target asset. |
| `trigger` | `{ notionalPrice, triggerType }` or `null` | `triggerType` is `upper` or `lower`. |
| `twapBucketCount` | `number` or `null` | TWAP total buckets. `0` or `null` means auto. |
| `completedFillCount` | `number` or `null` | TWAP executions completed. `null` on the snapshot, populated on updates. Clamp with `min(completedFillCount, twapBucketCount)`. |
| `placedAt` | `string` (ISO 8601) | |
| `closedAt` | `string` (ISO 8601) or `null` | |

Looking for a field that is not listed here? Fetch the full order from [`GET /orders/{orderId}`](/api-reference/flash/get-order).

## Heartbeats channel

Subscribe to `heartbeats` to receive a keep-alive frame once per second. This keeps the connection open when order updates are sparse, so a quiet stream is not closed for being idle. Subscribe with just your `apiKey`:

```json
{ "channel": "heartbeats", "type": "subscribe", "apiKey": "dpka_…" }
```

Each frame carries an incrementing `counter` and a timestamp:

```json
{ "channel": "heartbeats", "type": "heartbeat", "counter": 1, "timestamp": "2026-01-01T00:00:00.000Z" }
```

<Tip>
Subscribe to `heartbeats` alongside `orders` on every connection. It is the simplest way to keep a low-activity stream alive.
</Tip>

## Unsubscribing

Stop a channel with an `unsubscribe` message. The server re-acks your remaining channels.

```json
{ "channel": "orders", "type": "unsubscribe" }
```

## Example integration

A framework-agnostic reference in about 40 lines:

```ts
const WS_URL = "wss://flash.definitive.fi/v1/ws";
const API_KEY = "dpka_…"; // read-scoped Flash key
const FUNDER = "0x…";

const orders = new Map(); // orderId -> FlashOrderUpdate
const OPEN = new Set([
  "ORDER_STATUS_PENDING",
  "ORDER_STATUS_ACCEPTED",
  "ORDER_STATUS_PARTIALLY_FILLED",
]);

let attempts = 0; // grows on each failed reconnect, resets on a good connection

function connect() {
  const ws = new WebSocket(WS_URL);
  let stop = false; // set on a terminal error so we don't reconnect

  ws.onopen = () => {
    attempts = 0;
    ws.send(JSON.stringify({ channel: "orders", type: "subscribe", funderAddress: FUNDER, apiKey: API_KEY }));
    ws.send(JSON.stringify({ channel: "heartbeats", type: "subscribe", apiKey: API_KEY })); // keeps the socket alive
  };

  ws.onmessage = (e) => {
    const f = JSON.parse(e.data);
    if (f.type === "snapshot") {
      orders.clear();
      for (const o of f.orders) orders.set(o.orderId, o);
      render(orders);
    } else if (f.type === "update") {
      for (const o of f.orders) {
        if (OPEN.has(o.status)) orders.set(o.orderId, o); // replace by orderId
        else orders.delete(o.orderId); // drop terminal orders
      }
      render(orders);
    } else if (f.type === "error") {
      console.warn("[flash-ws]", f.code, f.message);
      if (f.code === "UNAUTHORIZED") { stop = true; ws.close(); } // bad key: fix it, don't retry
      // other errors leave the socket open; fall back to GET /orders if the stream stays down
    }
    // heartbeat and ack frames need no action
  };

  ws.onclose = () => {
    if (stop) return;
    const backoff = Math.min(30_000, 1_000 * 2 ** attempts++);
    setTimeout(connect, backoff / 2 + Math.random() * (backoff / 2)); // equal jitter; re-subscribes on reopen
  };
}
```

## Errors

Errors arrive as a frame you can act on:

```json
{ "type": "error", "code": "UNAUTHORIZED", "message": "apiKey does not match this connection" }
```

| Code | What it means | What to do |
| --- | --- | --- |
| `UNAUTHORIZED` | The `apiKey` is missing, invalid, or does not match this connection | Check your key. Do not reconnect. |
| `FUNDER_REQUIRED` | Subscribed to `orders` without a `funderAddress` | Include `funderAddress`. |
| `FUNDER_LOCKED` | Tried to stream a different funder on the same connection | Open a new connection per funder. |
| `CONNECTION_LIMIT` | Too many open connections for this key | Reduce concurrency for the key. A backed-off reconnect recovers as slots free. |
| `STREAMING_UNAVAILABLE` | The stream is temporarily unavailable | Fall back to [`GET /orders`](/api-reference/flash/list-orders). |

Malformed or transient errors (`BAD_JSON`, `UNKNOWN_CHANNEL`, `UNKNOWN_TYPE`, `SUBSCRIBE_FAILED`, `SNAPSHOT_FAILED`, `STREAM_ERROR`, `INTERNAL_ERROR`) mean the message was invalid or the server hit a temporary issue. Fix the message, or reconnect and re-subscribe.
