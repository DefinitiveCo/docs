---
title: Rate Limits
description: Per-endpoint request limits and 429 response behavior.
---

The Flash API enforces a rate limit of 5 requests per second on each endpoint, per API key:

| Limit      | Window     | Scope                      |
| ---------- | ---------- | -------------------------- |
| 5 requests | per second | Each endpoint, per API key |

Each endpoint (`quote`, `order`, list orders, get order, cancel) has its own independent bucket, so hitting the limit on one does not throttle the others.

<Note>
  Need higher throughput? We can support custom limits per client based on your needs. Let us know the rate limits you're looking for on each endpoint and we can set them up.
</Note>

### Response headers

Every response includes the following draft-6 rate-limit headers, scoped to the endpoint you called:

| Header                | Meaning                                            |
| --------------------- | -------------------------------------------------- |
| `RateLimit-Policy`    | Limit and window, e.g. `5;w=1`                     |
| `RateLimit-Limit`     | Maximum requests in the current window             |
| `RateLimit-Remaining` | Requests remaining in the current window           |
| `RateLimit-Reset`     | Seconds until the window resets                    |

On a `429` response, the same `RateLimit-*` headers are returned. Use `RateLimit-Reset` to know how many seconds until you can retry (at most 1, since the window is 1 second).

## Recommended client behavior

- Read `RateLimit-Remaining` after each successful response and throttle when it nears zero.
- On `429`, wait the number of seconds in `RateLimit-Reset` before retrying.
- Use exponential backoff with jitter for sustained pressure.
